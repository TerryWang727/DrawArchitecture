```
拆分电商系统为微服务
【背景】
假设你现在是一个创业公司的 CTO，开发团队大约30人左右，包括5个前端和25个后端，后端开发人员全部都是 Java，现在你们准备从0开始做一个小程序电商业务，请你设计微服务拆分的架构以及微服务基础设施选型。
【作业要求】
1. 需要明确服务拆分思路，并且将拆分后的系统架构图画出来；
2. 需要明确微服务基础设施选型思路，并选择一个微服务框架；
3. 用1~2页 PPT 即可。
【提示】
1. 需要应用三个火枪手原则；
2. 需要选择拆分方式；
3. 需要选择微服务框架的模式。
```

```
一、拆分思路

从0开始构建，所以微服务拆分根据业务拆分，搭建完善的基础设施，按照微服务基础设施优先级分级逐期落地，一步到位完全落地

借鉴阿里共享业务单元拆分逻辑
业务系统大概可以拆为：
1）用户服务
2）商品服务
3）交易服务
4）评价服务
5）店铺服务
6）搜索服务
7）数据服务
8）营销服务



由于团队规模为30人5个前端和25个后端，且后端开发人员全部都是 Java 开发
前端五人： C端3人、管理服务端2人
后端25人：拆分为8个服务交易管理较为复杂4个人，其余7个模块各三人
二、微服务基础设施选型思路

由于后端开发人员统一语言为java，前期技术栈比较统一，采取SDK方式更方便（原生高可用、高性能）。
这里我们选用spring cloud作为微服务开发框架
由于对spring cloud不熟悉 借鉴https://blog.csdn.net/w1014074794/article/details/88177857
1、eureka：服务注册发现、配置中心
2、zuul、gateway：服务网关、动态路由
3、es、logstash、kibana：日志系统
4、rabbitmq: 消息队列
5、redis:数据缓存
6、jenkins：自动化打包
7、docker：容器
8、k8s：应用部署、规划、更新、维护
9、promethus：服务监控、告警
10、sleuth：服务跟踪
11、hystrix:容错处理
12、Mysql：数据持久化
13、oss:对象存储
```

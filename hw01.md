```
作业二
假设今年学校毕业设计要求提升，要求做真正可运行的学生管理系统，学院对毕设的具体要求如下：

要求可以通过公网域名访问；
要求至少 3 人合作完成；
能够支撑管理 1000 个学生；
答辩的时候会根据架构方案来进行打分，不推荐太简单和太复杂的方案
你找了 2 个好朋友一起来做这个项目，你们的基本情况如下：

大家都会 Java，但是有一个是 PHP 高手
大家经济条件一般
作业要求：

对照面向复杂度架构设计方法论，构思 2 个以上的备选架构方案。
使用 PPT 来画出你的备选架构方案，并说明方案的优缺点。
给出你选择的最终方案以及选择理由。
```


```
面向复杂度架构设计方法论
本质：降低系统复杂度
思路：针对复杂的地方进行架构设计
模式：高性能、高可用、可扩展、安全、成本...
套路：分库分表、缓存、集群、分片、微服务、DDD、异地多活...
```

```
复杂度分析
1、性能分析
学生总数：1000  用户量级 并发较小
并发需考虑1000个操作/s

2、可用性考虑
学生系统实际运行或请求进来的时间仅为工作日和上课时间、所以系统运行时间估计约不到50%

3、可扩展
该系统后续维护，学校需求会随时代进步而提高要求，需注重扩展性和后续维护成本以及难以程度

4、安全
学生信息需要高度重视以及保密，需要私有化部署服务器

5、成本
用户量小，但是学生信息较为重要需要私有化部署
```

```
方案一：
基于白嫖云（阿里云、七牛云）等ecs做负载均衡
到本地由PHP高手写业务子系统 每个子系统由单独业务服务器处理

数据安全性：mysql每日本地备份+主从同步
```

```
方案二：
通过nginx做负载均衡到docker集群，分别去完成学生信息、课程等服务，最后数据通过mysql做主备
由于数据量较小不考虑用消息队列和缓存去做缓冲，若有需要可以加消息队列去缓冲负载

数据、服务之间不做隔离
数据安全性：mysql每日本地备份+主从同步
考虑到后期扩展自由性高、做微服务考虑到后期扩展需要高
```

```
方案三：
通过nginx做负载均衡
到本地由PHP高手写业务子系统 每个子系统由单独业务服务器处理
最后数据通过mysql做主备
数据安全性：mysql每日本地备份+主从同步
```

```
方案比较优缺点
成本：方案三>方案一>方案二
可扩展性：方案二相对扩展便捷
数据安全：方案一通过ecs做数据上云存在风险，方案二、三通过本地nginx做负载均衡，通过本地局域网做消息处理相对安全
开发效率：方案一、三能充分调动PHP高手，尽快完成系统开发


基于以上几点，从学生数据安全性考虑，后续功能可扩展，选择方案二，牺牲发效率和成本去完成本次架构
```
